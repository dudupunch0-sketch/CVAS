# CVAS Datapath Pipeline Design

## 1. Goal

CVAS is a C-model analysis tool for hardware-oriented datapath design.
Its primary output is a machine-readable JSON IR, and its secondary output is
a standalone HTML viewer for interactive inspection.

The first hardware target is:

- datapath-centric pipeline architecture
- control-flow aware
- initiation interval `II = 1`
- latency is allowed
- throughput should be one useful result per cycle after fill

## 2. System Boundary

CVAS should not be treated as a generic C compiler frontend.
It should instead act as an analysis pipeline that converts source code into
hardware-relevant facts:

- functions and call relations
- scalar and memory dependencies
- control flow and branch structure
- operation summaries and rough cycle cost
- pipeline candidates and hazards
- confidence and provenance for derived facts

The core CLI contract is deterministic and file-oriented. Optional semantic
metadata, such as function IO reads/writes, enters the main pipeline as an
explicit JSON file (`--function-io`) rather than as an implicit live LLM call.

## 3. Output Contract

### JSON

The JSON output is the canonical contract between analysis, CLI automation, and
the HTML viewer. Current outputs use additive Schema v3.

Top-level fields emitted by the current pipeline:

- `schema_version: "3.0"`
- `schema` metadata (`name`, `version`, compatibility flags)
- `blocks`
- `operations`
- `signals`
- `flow`
- `diagram_hint`
- `note`
- `analysis_version: "2.0"`
- `analysis_mode`
- `analysis_backend`
- `project_mode`
- `duplicate_functions`
- optional `gcc_dump` metadata in full mode. Its `command` is a normalized
  diagnostic display string (`command_path_policy: "normalized"`), so checked-in
  full-mode sample artifacts do not embed local temporary directories or
  checkout paths.

`schema_version` is the JSON contract version. `analysis_version` remains the
analyzer generation marker and should not be used by consumers to branch on the
JSON schema.

The v3 contract preserves these legacy v2 viewer fields:

- `blocks[]`
- `operations[]`
- legacy signal endpoint fields (`source_id`, `source_type`, `destination_id`,
  `destination_type`, `signal_name`, `direction`, `comment`)
- `flow.execution_order`
- `flow.parallelism`
- `flow.call_graph`
- `flow.call_sequence`

Schema v3 adds these implemented fields:

- `flow.execution_order_meta`: identifies `execution_order` as static block
  order, not a dynamic runtime trace.
- `flow.call_instances[]`: stable direct call occurrences with `call_id`,
  caller/callee block/function IDs, ordinal, argument metadata, optional return
  assignment, source text, and provenance. Repeated calls to the same callee are
  represented by different IDs such as `C_B2_0001` and `C_B2_0002`.
- enriched `signals[]`: optional semantic fields such as `signal_id`, `kind`,
  `role`, `call_id`, `arg_index`, `param`, `expr`, `target`,
  `source_function`, `destination_function`, and `provenance`.
- `flow.sequence_timeline[]`: viewer-ready timeline steps keyed to static block
  order, with caller/callee call IDs, incoming/outgoing signal IDs, and
  `read_write_summary` buckets.
- `flow.function_io`: normalized function reads/writes envelope. It is either
  imported from `--function-io` or generated by the rule-based fallback.
- `flow.dependencies`: indexes inter-block and call-instance dependencies by
  signal and call IDs.
- optional `flow.pipeline_stages.items[]`: viewer-consumed stage annotations
  keyed by `block_id` or `function`, with `stage`, optional `stage_label`, and
  optional `lane_role`. The Sequence viewer uses these annotations before
  falling back to stage-number function names.

Maintained contract docs and examples:

- `docs/schema/cvas.schema.v3.json`
- `docs/schema/cvas-schema-v3.md`
- `tests/fixtures/schema/sequence_timeline_v3.expected.json`
- `docs/test_examples_output_fast.json`
- `docs/test_examples_output_full.json`

Potential future extensions, not part of the current JSON contract unless added
explicitly later:

- `static_analysis`
- `llm_annotations`
- `merged_analysis`
- `pipeline`
- `confidence`
- `conflicts`
- `assumptions`

### HTML

The HTML output is a consumer of the JSON contract.
It should visualize the model, not re-derive semantics that are already present
in JSON.

Viewer responsibilities:

- Diagram view for block / operation / signal structure.
- Sequence view that prefers v3 `flow.sequence_timeline[]` and renders one
  execution board with selectable Call order, Dependency order, and Pipeline
  stage order layouts. Pipeline stage order groups stage-named functions such as
  `bpc_stage1_*` into shared stage columns with lane/helper rows. The board is a
  static analysis visualization, not a promised dynamic runtime or HLS schedule.
- Test-only Sequence import/export hooks are available when opening the viewer
  with `?test-hooks=1`; they expose DOM textareas/buttons for Browser or
  Playwright round-trip checks without relying on native file dialogs.
- Dense call IDs, signal IDs, function IO contracts, and provenance details stay
  in the Details panel rather than dominating the default Sequence board.
- Legacy Sequence fallback for v2-style JSON that only has `flow.call_sequence`.
- Visible call ID / signal ID details for provenance inspection.
- Local/offline operation with bundled assets.
- Stable rendering from generated JSON.

## 4. Analysis Architecture

### Stage A: Source Discovery

Responsibilities:

- extract the `CVAS_START` / `CVAS_END` region
- discover functions
- index project-level symbols when project mode is enabled
- normalize source enough for stable parsing

### Stage B: Static Analysis

Static analysis is the source of truth for deterministic facts.

Responsibilities:

- expression lowering
- operation classification
- control-flow extraction
- call graph construction
- call-instance extraction
- data-flow and call argument/return signals
- unresolved/external symbol reporting
- basic cycle estimation
- provenance for generated facts

### Stage C: Optional Function IO / Semantic Enrichment

The current main pipeline accepts semantic enrichment as a file artifact.
`tools/generate_function_io.py` can create `function_io.json` using a rule-based
path or an external LLM/agent-assisted path. `src/cvas_cli.py` imports that file
with `--function-io` and embeds a normalized envelope under `flow.function_io`.

Recommended uses for future enrichment:

- summarize function intent
- propose pipeline stage boundaries
- classify ambiguous memory intent
- annotate control intent
- suggest dependency candidates for review

LLM or agent output should not be the final authority for:

- exact data dependence
- aliasing decisions
- hazard existence
- cycle counts
- II=1 validity

Future CLI-agent handoff should keep CVAS responsible for producing static
facts, task/schema files, validation, provenance preservation, and coverage-gap
reporting. The external agent or internal model can perform semantic synthesis,
but imported results must remain explicit files that CVAS validates.

### Stage D: Merge and Validation

Merge static facts and optional imported annotations with static precedence.

Rules:

- static analysis wins on conflicts
- imported semantic output is treated as hypothesis unless validated
- final JSON preserves provenance and references stable IDs
- confidence should be explicit where available
- consumers should not invent call/signal identities in the viewer

### Stage E: Rendering

The HTML renderer consumes the JSON and provides:

- diagram rendering
- v3 Sequence execution-board rendering with call, dependency, and pipeline-stage layouts
- v2 Sequence fallback rendering
- manual inspection of provenance/confidence IDs
- offline usage with bundled assets

## 5. Hardware Interpretation

For the first target architecture, CVAS should answer:

- what the datapath is
- which operations can be staged in a pipeline
- where control decisions occur
- whether memory behavior threatens II=1
- what the likely critical path is
- which parts are unresolved and need user guidance

This is more useful than attempting a full RTL compiler on day one.

## 6. CLI Execution Model

The current CLI is batch-oriented and file-oriented:

- input C/C++ source file
- output JSON (`-o`)
- optional cycle-rule overrides
- optional project indexing (`--project-root`, `--entry-file`,
  `--source-extensions`)
- optional analysis mode selection (`--analysis-mode fast|full`)
- optional compile configuration in full mode (`--compile-arg`, `--compile-db`,
  `--language`; legacy `--clang-arg` and `--clang-compile-db` aliases remain
  accepted)
- optional `function_io.json` import (`--function-io`)

`src/cvas_cli.py` is the direct JSON-producing entrypoint. `src/cvas_mvp.py` is
a compatibility wrapper for legacy users. `cvas_wrapper.py` runs the JSON
analysis plus `json_to_html.py` and copies the offline viewer asset next to the
HTML output.

The user-facing CLI should remain deterministic: any optional enrichment should
be visible as an input file, output JSON section, and provenance-bearing data.

## 7. Refactoring Status and Next Phases

### Completed: Contract extraction and pipeline separation

- shared IR dataclasses live in `src/cvas_model.py`
- project indexing helpers live in `src/cvas_index.py`
- control-flow analysis lives in `src/cvas_cfg.py`
- call-graph analysis lives in `src/cvas_callgraph.py`
- expression lowering helpers live in `src/cvas_expr.py`
- serialization helpers live in `src/cvas_serialize.py`
- shared function-discovery helpers live in `src/cvas_source.py`
- shared statement/string helpers live in `src/cvas_text.py`
- CLI argument parsing / file I/O lives in `src/cvas_cli.py`
- `src/cvas_mvp.py` remains a thin compatibility wrapper
- function-level preprocessing/lowering lives in `src/cvas_passes.py`
- `src/cvas_pipeline.py` orchestrates model assembly

### Completed: Schema v3 sequence contract

- top-level `schema_version` / `schema` metadata
- stable repeated-call IDs in `flow.call_instances[]`
- enriched signal metadata and provenance
- `flow.sequence_timeline[]`
- normalized `flow.function_io`
- `flow.dependencies`
- viewer-first v3 Sequence execution-board rendering with v2 fallback
- schema docs, targeted tests, and checked-in sample artifacts

### Next: Pipeline scheduling

- add a pipeline model for `II = 1`
- represent control, data, and memory hazards explicitly
- use the model to explain where throughput is preserved or broken

### Next: Agent-assisted semantic enrichment

- keep external model/agent work as explicit file handoff
- define provider-neutral task/schema files
- validate imported annotations against stable block/call/signal IDs
- preserve provenance and coverage gaps in JSON

## 8. Open Questions

- Which C subset should be considered supported by default?
- How strict should the tool be when aliasing is unclear?
- Should the first release prioritize throughput analysis or control accuracy?
- What should be the minimum confidence threshold for imported semantic annotations?
- How much signal detail should the Sequence execution board show by default before filtering/search becomes necessary?
